# Minimal Sufficient Change

> version: 0.4.0

## 原则

目标是最小但充分的语义改动，不是最少行数。

优先级固定为：

1. 正确满足当前需求和既有契约；
2. 保留安全、数据完整性和必要错误处理；
3. 清晰、可维护、符合项目风格；
4. 修改范围尽可能小；
5. 代码量尽可能少。

## 流程

1. 确定行为：用一句话说明要改变的可观察行为和完成条件。只有当歧义会影响公共行为、API、数据、安全或破坏性操作时才提问；否则采用最窄、最符合现有代码的解释，并说明假设。

2. 先读再写：阅读目标代码、调用方、相关测试、项目规则、已有 helper 和已有依赖。先搜索再创建，不把任务扩大成全库审计。

3. 按阶梯实现：优先顺序为：

   不写新代码或只调调用方式
   → 复用项目已有代码
   → 标准库、语言原生或平台能力
   → 已安装且合适的依赖
   → 现有文件中的局部修改
   → 新 helper、文件、抽象或依赖。

4. 放在正确边界：缺陷来自共享逻辑时，在共享所有者处修；行为只属于单一路径时，局部修。最小文本 diff 放在错误层次，也不是最小正确改动。

## 范围约束

每一处改动都必须能追溯到当前需求、既有契约、必要接入、必要验证，或本次改动直接造成的清理。

不要顺手重命名、格式化、改无关注释、整理目录、清历史债务或重构相邻代码。

## 膨胀禁区

没有当前、具体、可说明的需要时，不要新增：

- 未要求的功能；
- 为未来预留的配置、环境变量、feature flag、可选参数或分支；
- 只有一个实现的 interface、factory、strategy、registry 或基类；
- 只转发调用的 wrapper；
- 单次使用且没有清晰领域含义的 helper；
- 为少量普通逻辑增加的新依赖；
- 放在现有文件仍然清晰时的新文件；
- 未使用的 public API、字段、事件、类型或导出；
- 占位代码、脚手架或“以后再用”的 TODO 分支。

少量清晰重复可以保留。共同语义稳定后再抽象。

例外：真实的外部系统边界、信任边界、事务边界或关键领域不变量，即使当前只有一个实现，也可以建立清晰边界。

## 安全底线

不得为了少代码删除或削弱：

认证授权、输入校验、转义清洗、secrets 处理、防数据损坏、必要错误处理、事务、并发保护、幂等性、信任边界校验、基本可访问性、用户明确要求的失败行为。

不要为被可靠契约、类型系统或数据库约束排除的状态重复防御；但外部输入、网络、文件、环境变量和持久化数据都应视为可能异常。

不要写聪明但难读的一行式代码。清晰的 15 行通常好过脆弱的 3 行。

## 验证

验证与风险相称：

- bug 修复优先加回归测试；
- 非平凡逻辑覆盖主路径、关键边界和相关失败路径；
- 简单接线可用现有测试、类型检查、构建或最小运行检查证明；
- 安全、资金、迁移、删除、并发和数据完整性路径不得被“少测试”限制。

不要削弱测试、删除失败断言、硬编码结果，或修改测试来迎合错误实现。

## 交付前 diff 审查

只审本次 diff：

- 能否删除新增代码、文件、依赖、公共 API、参数、配置或抽象？
- 是否重复了项目已有能力、标准库或平台能力？
- 每个新增概念现在是否有价值？
- 修改是否在正确所有权边界？
- 是否混入无关重构、格式化、重命名或注释修改？
- 收缩后是否仍正确、安全、清晰、可验证？

能安全删除就删除，重新运行相关检查，然后停止。不要为了“更优雅”继续重构。

## 交付说明

只说明：

- 改了什么；
- 如何验证；
- 真实风险或限制。

不要把“减少了多少行”或“只改了几个文件”当作成功证明。
