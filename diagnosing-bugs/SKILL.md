# 诊断 Bug

这是一套用于快速诊断并修复棘手 Bug 的工作纪律。速度来自短反馈闭环、必要的上下文理解和清晰证据链，而不是跳过验证、提前下结论或先写补丁。目标是复现问题、收敛变量、验证根因，并交付可回归验证的最小修复。

在阅读或修改代码前，先查找并阅读与问题模块相关的已有上下文，例如 `CONTEXT.md`、设计说明、ADR、release note、issue、变更记录、最近相关提交，或仓库自己的运行/交接文档。不要只看报错附近几行，也不要泛读无关模块；你需要理解本次问题涉及模块的整体思路、主要流程、设计意图、关键状态、上下游依赖、历史兼容约束，以及被怀疑代码为什么存在。

## 硬规则

- 没有诊断闭环，不要宣称 root cause。
- 没有理解相关模块的流程、设计意图和影响面，不要判定代码错误或提交修复。
- 假设必须有事实来源，并且可以被验证或排除。
- 一次只验证一个假设，一次只引入一个会改变行为的修改。
- 任何被证明无效的行为修改、临时补丁或配置改动，必须立即删除或回滚；绝不能在无效修改之上继续验证下一个猜想。
- 修复必须追溯到已验证的失败机制。
- 临时 workaround 不能冒充正式修复。
- 修完必须重新验证原始问题，并确认没有破坏已知正常路径。

## 强制门禁

使用本 skill 时，维护一个显式阶段状态清单，不能只在内部记忆中跟踪。状态只能使用：`未开始`、`进行中`、`通过`、`部分通过`、`阻塞`。当任务跨越多个回合、被用户质疑流程、或准备最终交付时，必须更新这份清单。

进入修复前，先写清楚阶段 1 的红/绿闭环：

- 失败特征是什么；
- 什么条件下会出现；
- 修复后用什么方式证明同一症状消失。

只要以下任一项未满足，不得使用“完成”“都做完了”“已解决”等完成类措辞：

- 原始红/绿闭环未通过；
- known-good 路径未验证，或未明确说明无法验证；
- 临时修改、调试配置、一次性脚本或运行时状态未清理，且未明确说明保留原因；
- 阶段 6 交付字段未完整填写；
- 未覆盖项和剩余风险未明确列出。

当用户询问“是否严格按步骤”“哪一步没做”“为什么没遵守”“有没有按 skill”这类流程合规问题时，停止继续推进技术细节，先做合规差距审计。审计必须用表格列出：

| 阶段 | skill 要求 | 已有证据 | 状态 | 缺口 |

## 阶段 1：建立最短诊断闭环

先建立能区分“问题存在 / 问题消失”的最短诊断闭环。闭环是这项技能的核心；没有闭环，后续判断很容易变成猜测。

诊断闭环不一定是自动化测试。它可以是：

- 一个最小复现步骤；
- 一组固定输入和输出对比；
- 一条能稳定暴露失败特征的日志、dump 或 trace；
- 一个 known-good / known-bad 版本、配置或数据对比；
- 一个可以重复执行的设备、环境或实验室验证步骤；
- 一个局部 harness，用来重建可疑代码路径。

常见构造方式可以包括：

1. **用户提供的复现输入或步骤**：把用户给出的现象转成可重复执行的最短动作。
2. **失败测试**：选择能触达该 Bug 的测试切入点，可以是单元测试、集成测试或端到端测试。
3. **CLI 调用**：使用固定样例输入运行命令，并把 stdout/stderr、退出码或产物与 known-good 快照做对比。
4. **日志 / dump / trace 回放**：保存能代表真实失败的日志、dump、trace、事件序列或输入样本，再在隔离路径中回放或复查。
5. **局部 harness**：在可疑模块、函数、脚本、驱动入口或设备命令附近搭一个最小调用路径，重建可疑代码路径。
6. **性质测试 / 模糊测试循环**：如果 Bug 表现为“有时输出错误”，运行大量随机或边界输入寻找失败模式。
7. **二分定位脚手架**：如果 Bug 出现在两个已知状态之间，自动化“切到状态 X、启动、检查、重复”，让它可以交给 `git bisect run`。
8. **差分验证闭环**：用同一份输入分别跑旧版本和新版本、两套配置或两组数据，再比较输出。
9. **HITL Bash 脚本**：最后手段。如果必须有人操作，用 `scripts/hitl-loop.template.sh` 引导人工步骤，并捕获结构化输出。

闭环必须回答三个问题：

1. 失败特征是什么？
2. 什么条件下会出现？
3. 修复后用什么方式证明它消失？

闭环必须能判定用户具体症状的存在/消失，不能只是证明“看起来正常”或“跑完没报错”。红必须代表同一个具体症状存在，绿必须代表同一症状已经消失。

如果暂时没有可运行环境，就基于已有日志、dump、trace、版本差异或配置差异建立 artifact-driven 闭环。此时要明确已有证据能证明什么、还缺什么，以及下一项最能缩小范围的证据是什么。

阶段 1 的完成标准：你能给出一条具体命令、固定步骤或证据判定方式，并且已经至少执行或检查过一次。它必须命中用户描述的具体问题，而不是只证明“程序能跑”或“系统没有崩溃”。如果需要人工参与，只允许通过 `scripts/hitl-loop.template.sh` 这类结构化 HITL 闭环承接。

## 阶段 2：复现并收敛变量

运行阶段 1 的闭环，确认观察到的失败就是用户描述的问题，而不是旁边另一个相似问题。

捕获精确症状：错误信息、异常输出、失败日志、性能退化、复现概率、触发条件，以及首次出现问题的版本或条件。

然后收敛变量。重点不是“复现一次”，而是缩小到能指导定位的最小差异：

- 哪个版本、配置、输入、数据或改动开始异常；
- 哪个运行条件、外部条件、操作顺序或状态转换相关；
- known-good 与 known-bad 之间的最小差异是什么；
- 去掉哪些条件后问题消失，保留哪些条件后问题仍然存在。

每次只改变一个变量。对于偶现问题，目标不是一次复现，而是提高复现率：循环执行、增加压力、固定条件、并行运行或缩小时序窗口，直到它足以支撑调试。

在最小差异尚不清楚前，不要急着修改代码。

## 阶段 3：基于上下文提出可证伪假设

阶段 3 的目的不是凭空猜原因，而是基于已观察到的症状、最小差异、日志、配置、版本信息，以及对相关模块流程和设计意图的理解，提出可验证的候选原因。

在列出假设前，先建立对问题相关模块的足够理解：

- Bug 从哪里进入；
- 经过哪些关键流程、状态转换或系统边界；
- 失败点上下游的调用关系；
- 相关配置、版本、数据或外部条件；
- 被怀疑代码原本的设计意图、兼容职责或保护作用。

不要把尚未理解的代码直接判定为错误。一个好的假设应该解释“为什么这段原本合理的逻辑，在当前条件下导致了失败”。

通常列出 2 到 5 个按可能性排序的假设。每个假设必须包含：

- 支持它的已观察事实；
- 涉及的代码路径或系统边界；
- 怀疑的失败机制；
- 可观察预测；
- 验证或排除它的方法。

推荐格式：

> 基于 ____ 现象，我怀疑 ____ 路径中的 ____ 机制导致问题。
> 如果它是真的，那么 ____ 应该出现 / 消失 / 变得更严重。
> 我将通过 ____ 来验证。

如果一个假设没有事实来源，或者不能导出预测，它只是感觉，不足以进入插桩或修复。

## 阶段 4：验证假设并定位根因

每一次插桩、实验或对比都必须对应阶段 3 中的某个具体预测。一次只验证一个假设，一次只改变一个变量。

优先使用能直接区分假设的观测方式：debugger、trace、定向日志、状态 dump、版本差分、配置差分、输入输出差分、性能基线或 profiler。不要“把所有东西都打日志，然后再 grep”。

区分两类临时改动：

- **观测型插桩**：只增加日志、trace、dump 或断言，不改变系统行为。必须有唯一标记，例如 `[DEBUG-a4f2]`，方便最后清理。
- **行为型修改**：会改变逻辑、时序、配置、默认值、返回值或错误处理路径。每次只能引入一个，用来验证一个具体预测。

如果行为型修改证明无效，必须立即回滚到干净基线，再验证下一个假设。不要叠加多个无效修改；不要在失败实验留下的代码上继续修改。

对于性能问题，先测量，再修复。先建立基线、热点位置和退化幅度，再判断原因。

阶段 4 的输出应该是以下两种之一：

- 某个假设被证据验证，并能解释失败机制；
- 当前假设被排除，相关行为修改已回滚，需要带着新证据回到阶段 3。

如果证据只能说明某个 workaround 有效，但不能解释为什么有效，不要把它当成 root cause。

## 阶段 5：基于已验证根因修复

只有当某个假设已经被阶段 4 的证据验证，才进入修复。修复必须能追溯到这个已验证的失败机制。

如果修改过程中发现原假设不成立，或者需要依赖新的解释，停止修改，完整回滚当前无效修改，再回到阶段 3 或阶段 4。不要在无效补丁上继续叠加新补丁。

修改代码前，先理解被修改代码的原设计意图、调用方、兼容约束和影响面。不要因为某段代码触发了当前 Bug，就直接删除或重写它；它可能正在保护其他版本、配置、运行路径或异常场景。

修复原则：

- 做最小修改；
- 修改必须可解释、可回滚；
- 不顺手重构；
- 不改变无关路径；
- 优先保留兼容路径，用 guard、条件约束、版本判断或局部修正解决具体问题；
- 不把 workaround 冒充 root cause fix。

如果存在正确的测试边界，先把最小复现固化为失败测试，再修复，并亲眼看它从失败变为通过。

如果不存在正确的测试边界，这本身也是发现。记录原因，并保留最接近真实问题的验证方式，例如诊断脚本、harness、trace replay、版本对比或固定复现步骤。

修复后必须重新运行阶段 1 的原始诊断闭环，并验证 known-good 路径没有被破坏。

## 阶段 6：验证、交付与复盘

宣布完成前，必须满足：

- 原始问题闭环已经通过；
- 回归测试或替代验证已经通过；
- known-good 路径没有被破坏；
- 无效修改、临时补丁、调试配置和一次性脚本已删除或回滚；
- 临时日志和探针已清理，或明确标记为调试用途；
- root cause、验证证据、修复方式、影响范围和剩余风险已经记录。

最终回复前，必须逐项检查 Stage 6 门禁：

- `[ ]` 原始问题闭环通过；
- `[ ]` 回归测试或替代验证通过；
- `[ ]` known-good 路径没有被破坏，或无法验证的原因已明确；
- `[ ]` 无效修改、临时补丁、调试配置和一次性脚本已删除或回滚，或保留原因已明确；
- `[ ]` 临时日志和探针已清理，或明确标记为调试用途；
- `[ ]` 未覆盖项和剩余风险已列出；
- `[ ]` 复盘已写。

交付结论必须包含以下字段，字段不可省略：

- 问题现象；
- 影响范围；
- 根因；
- 修复方案；
- 验证方式；
- known-good 路径；
- 任何临时措施的撤销或替换说明；
- 未覆盖项；
- 剩余风险；
- 复盘。

最后复盘：什么本可以防止这个 Bug 出现？

如果答案指向测试边界缺失、架构耦合、配置管理混乱、版本管理不清、日志不足或流程缺陷，把它作为后续改进项记录下来。不要在根因修复完成前借机大范围重构。
